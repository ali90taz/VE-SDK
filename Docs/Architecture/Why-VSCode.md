<h2 align="center">
Why VS Code?
<br>
<br>
</h2>

<h3>What is this document?</h3>

<h4 align="left">
This document explains why the current <b>VE-SDK</b> ecosystem is being 
developed using <b>Visual Studio Code</b> during the early stages of VitaEngine.
<br>
<br>
Its purpose is to clarify a practical development decision and prevent it from 
being misread as a statement about the final identity of the VitaEngine platform.
<br>
<br>
</h4>

---

<h3>Short answer</h3>

<h4 align="left">
<b>VE-SDK is currently being developed using VS Code because it is a practical, 
mature and efficient environment for building VitaEngine during its pre-alpha 
phase.</b>
<br>
<br>
This is a decision about <b>how VE-SDK is currently being built</b>.
It is <b>not</b> a statement about what <b>VitaEngine itself</b> is intended to 
become.
<br>
<br>
</h4>

---

<h3>What this means — and what it does not mean</h3>

<h4 align="left">
Using <b>VS Code</b> in the current VE-SDK workflow means:
<br>
<br>
<ul>
  <li>VE-SDK benefits from a mature development environment</li>
  <li>Development friction is reduced during the project’s early stages</li>
  <li>The project can move faster without rebuilding solved infrastructure too early</li>
</ul>
<br>
It does <b>not</b> mean:
<br>
<br>
<ul>
  <li>VitaEngine is intended to become a VS Code-shaped product</li>
  <li>VitaEngine is intended to inherit VS Code as part of its final identity</li>
  <li>VitaEngine is intended to carry VS Code concepts into its long-term user-facing experience</li>
  <li>VE-SDK’s current host environment defines the final platform vision</li>
</ul>
<br>
This distinction is important.
<br>
<br>
</h4>

---

<h3>VE-SDK and VitaEngine are not the same thing</h3>

<h4 align="left">
At the current stage of the project, it is important to separate:
<br>
<br>
<ul>
  <li><b>VE-SDK</b> → the technical development repository and ecosystem used to build VitaEngine</li>
  <li><b>VitaEngine</b> → the long-term platform being created through that ecosystem</li>
</ul>
<br>
Because of that:
<br>
<br>
<ul>
  <li><b>VE-SDK</b> may use strong existing tools to accelerate development</li>
  <li><b>VitaEngine</b> does not need to inherit those tools as part of its final product identity</li>
</ul>
<br>
In simple terms:
<br>
<br>
<b>VE-SDK may be built with the help of VS Code.</b>
<br>
<b>VitaEngine itself is not intended to be defined by VS Code.</b>
<br>
<br>
</h4>

---

<h3>Why VS Code makes sense right now</h3>

<h4 align="left">
At this stage, VS Code offers several advantages that make it especially useful 
for developing VE-SDK:
<br>
<br>
<ul>
  <li><b>Fast iteration</b> for JavaScript, Electron, Lua, C/C++ and scripting work</li>
  <li><b>Mature extension ecosystem</b> across multiple languages and workflows</li>
  <li><b>Integrated terminal</b> for setup, scripts and build commands</li>
  <li><b>Strong Git integration</b> for a multi-branch project workflow</li>
  <li><b>Low friction</b> for technical notes, documentation and architecture work</li>
  <li><b>Excellent fit</b> for a project that mixes code, scripts, assets and documentation</li>
</ul>
<br>
In practical terms, it is one of the most efficient environments available for 
building a project that is simultaneously:
<br>
<br>
<ul>
  <li>a desktop application</li>
  <li>a tooling ecosystem</li>
  <li>a script-heavy SDK</li>
  <li>a future engine/runtime platform</li>
</ul>
<br>
</h4>

---

<h3>Why not replace VS Code immediately?</h3>

<h4 align="left">
Trying to replace a mature editor environment too early would create unnecessary 
cost and friction during pre-alpha.
<br>
<br>
Doing that now would likely mean:
<br>
<br>
<ul>
  <li>More maintenance burden</li>
  <li>Slower iteration</li>
  <li>More time spent recreating solved infrastructure</li>
  <li>Less focus on the parts of VitaEngine that are actually unique</li>
  <li>Higher risk of premature architectural lock-in</li>
</ul>
<br>
At this stage, the goal is not to prove independence by rejecting good tools.
The goal is to <b>make VitaEngine real</b>.
<br>
<br>
</h4>

---

<h3>What VitaEngine should focus on instead</h3>

<h4 align="left">
The unique value of VitaEngine does not come from replacing text editing itself.
<br>
<br>
Its real value is expected to come from building a <b>PS Vita-focused creation 
ecosystem</b>, including areas such as:
<br>
<br>
<ul>
  <li>Project structure and <b>.vep</b> workflow</li>
  <li>PS Vita-oriented project management</li>
  <li>Real hardware preview and iteration</li>
  <li>Companion integration</li>
  <li>Packaging and deployment flow</li>
  <li>A higher-level API designed around Vita creation</li>
  <li>A more direct path from idea to runnable software on the console</li>
</ul>
<br>
These are the parts that matter most.
<br>
<br>
VS Code is simply a strong host environment while those parts are still being 
built.
<br>
<br>
</h4>

---

<h3>Why "using VS Code" is not the same as "being based on VS Code"</h3>

<h4 align="left">
The current VE-SDK workflow should not be interpreted as:
<br>
<br>
<ul>
  <li>“VitaEngine is a VS Code extension”</li>
  <li>“VitaEngine is meant to remain attached to VS Code”</li>
  <li>“VitaEngine will carry VS Code as part of its long-term user-facing model”</li>
</ul>
<br>
A more accurate interpretation is:
<br>
<br>
<ul>
  <li>VS Code is the current <b>host environment</b> used to build VE-SDK</li>
  <li>VE-SDK is the evolving <b>technical ecosystem</b> used to create VitaEngine</li>
  <li>VitaEngine is the long-term <b>platform</b> being built through that process</li>
</ul>
<br>
This means VS Code is part of the <b>construction context</b>, not part of the 
<b>final platform identity</b>.
<br>
<br>
</h4>

---

<h3>This does not define VitaEngine’s final identity</h3>

<h4 align="left">
The current use of <b>VS Code</b> should be understood strictly as a 
<b>development-stage decision for VE-SDK</b>.
<br>
<br>
It exists because VE-SDK needs a practical, mature and efficient environment for 
building the VitaEngine ecosystem during its early phases.
<br>
<br>
This should <b>not</b> be interpreted as a statement that the final 
<b>VitaEngine</b> platform is intended to inherit, depend on, or conceptually 
carry parts of the VS Code environment.
<br>
<br>
At least in the current vision, <b>VitaEngine itself is not intended to be a VS 
Code-shaped product</b>.
<br>
<br>
The role of VS Code here is purely instrumental:
<br>
<br>
<ul>
  <li>to accelerate VE-SDK development</li>
  <li>to reduce unnecessary friction during pre-alpha</li>
  <li>to avoid rebuilding solved infrastructure too early</li>
</ul>
<br>
In short:
<br>
<br>
<b>VS Code is part of how VE-SDK is currently being built.</b>
<br>
<b>It is not intended to define what VitaEngine itself should become.</b>
<br>
<br>
</h4>

---

<h3>Final note</h3>

<h4 align="left">
VitaEngine should not try to prove itself by rejecting strong existing tools too 
early.
<br>
<br>
It should prove itself by building the parts that are genuinely worth building.
<br>
<br>
Right now, <b>VS Code is part of the scaffolding that helps VE-SDK build 
VitaEngine</b>.
<br>
<br>
That scaffolding should not be confused with the final shape of the platform.
<br>
<br>
</h4>
